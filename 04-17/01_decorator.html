1. 왜 데코레이터를 배우는가

데코레이터는 "함수를 입력받아 함수를 돌려주는 함수"이다. 한 줄 정의지만 파이썬 표현력의 중심축이다. 실무에서 데코레이터가 중요한 이유는 크게 셋이다.

  • 횡단 관심사 분리 — 로깅·인증·캐시·타이밍 같은 공통 기능을 본 로직을 수정하지 않고 덧붙인다.
  • 등록 패턴 — 여러 함수를 하나의 레지스트리에 자동 등록(라우팅, 이벤트 핸들러).
  • 프레임워크 가독성 — Flask @app.route, pytest @pytest.fixture, FastAPI @app.get, Click @click.command 같은 API의 전부가 데코레이터다.
이 노트북의 목표 수작업으로 "함수를 모으는" 코드를 먼저 작성해 본 다음, 그 동일한 일을 @ 문법당으로 간결하게 표현하는 과정을 눈으로 확인하는 것.

2. 셀 단위 해설

Cell 0Markdown
Decorator · 함수가 다른 함수를 불러온다 · 1. 수작업
무엇을: 데코레이터의 핵심 직관을 한 줄로 요약하고, 이어질 절을 "1. 수작업"으로 열어간다.
왜: 강의는 의도적으로 문법당(@)을 먼저 쓰지 않는다. @는 사실 문법의 껍데기일 뿐이고, 본질은 "함수를 다른 함수에 넘긴다"이기 때문에 먼저 원시적인 방식을 경험시키려는 교육적 설계이다.
Cell 1 — 수작업 수집기Code
# 수집된 함수들
collected_functions = []

def collect(func):          # "수집하다" — 데코레이터의 동작을 직관적으로 표현
    print(f'collecting: {func.__name__}')
    collected_functions.append(func)
    return func

def f1():
    print('running f1()')

collect(f1)
무엇을 하는지 · collect는 임의의 함수 객체를 인자로 받아 ① 그 이름을 찍고, ② 글로벌 리스트에 저장하고, ③ 동일한 함수 객체를 그대로 반환한다. 그 뒤 f1을 정의하고 수작업으로 collect(f1)을 호출해 리스트에 넣는다.
왜 이렇게 쓰는지 · 파이썬에서 함수는 일급 객체(first-class object)이기 때문에 다른 변수처럼 전달·저장·반환이 가능하다. func.__name__은 함수의 고유 속성 — 함수가 객체라서 속성이 붙어 있다는 결정적인 증거다. return func이 있어야 이후 데코레이터 문법(f1 = collect(f1))에서 f1이 사라지지 않고 그대로 살아남는다.
[파이썬 함수는 객체다] f1 ─── 이름 바인딩 ───▶ <function object at 0x...> ↑ collect(f1)이 이 객체 자체를 받아 리스트에 넣는다
Cell 2 — 수집 확인Code
collected_functions
무엇을: 현재까지 수집된 함수 리스트를 출력.
왜: REPL/Jupyter에서 마지막 표현식은 자동으로 repr로 찍힌다. 여기서 출력은 [<function f1 at 0x...>] 형태의 한 줄이 나온다 — "리스트 안에 함수 객체 그대로가 들어있다"는 핵심을 육안으로 확인시켜 준다.
Cell 3 — 두 번째 함수 수동 등록Code
def f2():
    print('running f2()')

collect(f2)
무엇을: f2를 새로 정의하고 수동으로 collect에 넘긴다.
왜 이 단계가 필요한가: 같은 작업의 반복을 보여주기 위함이다. 함수를 여러 개 수집해야 할 때마다 collect(fN)을 직접 불러주는 것은 번거롭고 잊어버리기 쉽다. 바로 이 "지루함"이 다음 절에서 @를 도입하는 동기가 된다.
Cell 4 — 수집 리스트 재확인Code
collected_functions
출력 예시: [<function f1 at 0x...>, <function f2 at 0x...>] — 두 함수가 같은 리스트에 나란히 저장되어 있다.
Cell 5Markdown
2. Decorator(@)
이제 동일한 일을 @ 문법당으로 다시 한다. 코드 양과 호출 구조의 차이를 앞 절과 직접 비교하는 것이 핵심.
Cell 6 — @ 데코레이터 등록Code
registry = [] # 함수들이 저장될 레지스트리 (리스트)

def register(func):
    # 모듈 로드 시 이 부분이 즉시 실행됩니다.
    print(f'running register({func})')
    registry.append(func)
    return func # 장식된 함수를 그대로 반환합니다.

@register
def f1():
    print('running f1()')

@register
def f2():
    print('running f2()')

@register
def f3():
    print('running f3()')

print(f'Registry after loading module: {registry}')
무엇을: register는 앞의 collect와 시그니처가 똑같다(함수 받아 리스트에 넣고 반환). @register를 세 번 쓰니 f1/f2/f3이 정의되는 즉시 자동으로 등록된다. 마지막 print는 레지스트리에 세 함수가 모두 담겨 있음을 보여준다.
왜 이렇게 쓰는지: @register는 정확히 f1 = register(f1)의 문법당(syntactic sugar)이다. 시각적으로 "이 함수는 등록 대상"이라는 의도가 함수 선언 바로 위에서 드러나므로, 유지보수자가 구조를 파악하기 쉽다.

실행 순서 (중요)

  1. 파이썬이 모듈을 위에서 아래로 읽는다.
  2. registry = []로 빈 리스트를 만든다.
  3. register 함수 정의가 끝난다.
  4. @register 아래 def f1(): ... 블록을 만난다 → 내부적으로 f1 = register(f1)이 실행되고 그 안에서 print('running register(<f1>)')가 즉시 찍힌다.
  5. f2, f3에 대해 동일.
  6. 마지막 print가 실행되어 리스트 세 개가 표시된다.
핵심 관찰 f1()을 호출하지 않아도 register의 본문이 이미 실행된다는 점이다. 데코레이터는 정의 시점에 동작하는 코드다. 이것이 "함수를 호출하기 전에 등록부터 끝내 놓는" Flask·FastAPI 라우팅의 기본 원리다.

연습 공간. 학생이 직접 데코레이터를 하나 더 만들어 보도록 비워 둔 자리이다.

3. @ 문법당의 내부 동작

두 표현은 완전히 동일하다. 데코레이터를 이해한다는 것은 두 줄이 서로 바꿔 쓸 수 있다고 확신하는 것이다.

# 데코레이터 문법
@register
def greet():
    print("hi")

# 위와 정확히 같은 의미
def greet():
    print("hi")
greet = register(greet)

파이썬은 이 두 표현을 같은 바이트코드로 번역한다. 따라서 @는 "멋 내기"가 아니라 가독성을 위한 공식적인 재작성 규칙일 뿐이다.

4. 클로저 · first-class 함수와의 관계

본 노트북은 "인자 변경 없이 그냥 통과"하는 가장 단순한 데코레이터만 다룬다. 그러나 실무에서는 동작 자체를 감싸는 래퍼가 훨씬 많다. 그때 등장하는 개념이 클로저다.

import time
from functools import wraps

def timing(func):
    @wraps(func)                     # 원본 __name__/__doc__ 보존 — 뒤에서 설명
    def wrapper(*args, **kwargs):   # 클로저: 바깥의 func을 포획
        t0 = time.perf_counter()
        result = func(*args, **kwargs)
        dt = time.perf_counter() - t0
        print(f'{func.__name__} took {dt:.4f}s')
        return result
    return wrapper

@timing
def slow_add(a, b):
    time.sleep(0.1)
    return a + b

slow_add(1, 2)   # ▶ slow_add took 0.1001s
차이를 정리 "등록형" 데코레이터(노트북의 register)는 함수를 그대로 반환한다. "래퍼형" 데코레이터(timing)는 새 함수 wrapper를 반환한다. 두 패턴을 구분해 두면 다른 코드를 읽을 때 혼란이 크게 준다.

5. 실무 패턴 5가지

패턴대표 예핵심
등록 (registry)Flask @app.route('/')함수를 URL → 핸들러 매핑에 넣는다
타이밍/로깅@timing, @log_calls입력·실행시간·출력 관측
캐시 (memoize)functools.lru_cache인자별 결과를 저장해 재호출을 생략
권한 체크@login_required전처리에서 인증 미달이면 예외/리디렉션
재시도@retry(times=3)실패 시 자동 재시도 (인자 받는 데코레이터)

마지막 "인자 받는 데코레이터"는 3단 구조(def outer(cfg): def deco(f): def wrapper(*a,**kw): ... return wrapper; return deco; return outer)가 되어 한 번에 이해가 안 될 수 있다. 이 노트북의 범위를 넘지만, 원리는 동일하다: "호출 시점에 함수를 감싸서 반환한다."

6. 흔한 함정과 functools.wraps

  • 메타데이터 유실: 래퍼형 데코레이터를 씌우면 func.__name__, __doc__, 시그니처가 전부 wrapper의 것으로 바뀐다. 디버깅·문서화 툴이 오작동한다. 해결: @functools.wraps(func)로 원본 메타데이터 복사.
  • 함수 객체 ≠ 호출 결과: collect(f1())처럼 괄호를 붙이면 함수를 실행한 결과(보통 None)가 인자가 된다. collect(f1)처럼 괄호 없이 넘겨야 함수 객체 자체가 전달된다.
  • 전역 상태 오염: 본 노트북의 registry는 전역 리스트다. 실제 서비스에서 전역 상태는 테스트·병렬 실행에 위험하므로 모듈 레벨 객체·클래스 속성으로 감싸는 편이 안전하다.
  • 적용 순서: 데코레이터 여러 개를 쌓으면 아래에서 위로 적용된다. @A 위에 @B면 B(A(func)) 순.
