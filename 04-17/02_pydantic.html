1. 왜 Pydantic을 배우는가

파이썬의 타입 힌트는 실행 시점에 강제되지 않는다. 즉 id: int라고 써놓아도 누군가 문자열을 넣어버리면 그대로 흘러간다. 작은 스크립트면 상관없지만 API·설정 로더·CLI 파라미터처럼 외부에서 들어오는 데이터를 다룰 때는 이 허술함이 치명적이다.

Pydantic은 "타입 힌트를 실제 검증 규칙으로 승격시키는" 라이브러리다. 선언한 필드와 조건을 위반하면 ValidationError를 던지고, 변환 가능한 타입은 자동으로 강제 변환한다. 이 특성 덕분에 FastAPI는 item: Item이라는 타입 힌트 한 줄만으로 요청 JSON의 파싱·검증·422 에러 응답을 전부 자동화한다.

핵심 공식 dataclass = "타입은 힌트일 뿐" / Pydantic BaseModel = "타입은 규칙이다"

2. 셀 단위 해설

Cell 0 — dataclass는 런타임 검사를 안 한다Code
import dataclasses

## 1. dataclass 예제
@dataclasses.dataclass
class DataUser:
    """타입 힌트를 사용하지만, 런타임 유효성 검사는 없음."""
    id: int
    name: str
    is_active: bool = True

# (1) 올바른 데이터
user_data = DataUser(id=1, name="Alice")
print(f"✅ 유효한 데이터 (정상): {user_data}")

# (2) 유효하지 않은 데이터 (str 대신 int를 할당)
# dataclass는 오류를 발생시키지 않고 값을 그대로 저장함
invalid_user = DataUser(id=2, name=123)
print(f"⚠️ 유효하지 않은 데이터 (저장됨): {invalid_user}")
무엇을: @dataclasses.dataclass로 3개 필드를 가진 DataUser를 정의하고, 잘못된 타입(name이 정수 123)을 일부러 넣어 본다.
왜 이렇게 쓰는지: 학생이 첫 번째로 깨달아야 할 사실이 "파이썬 타입 힌트는 기본적으로 검사하지 않는다"이기 때문이다. DataUser(id=2, name=123)은 에러 없이 통과한다. 즉 객체 속 name은 정수 123을 그대로 갖게 된다. 이것이 스크립트 레벨에서는 어떤 부작용을 만드는지 스스로 관찰하게 한다.
dataclass의 정체 dataclass는 "__init__, __repr__, __eq__를 자동으로 써주는 도우미"일 뿐이다. 타입 힌트는 그 자체로 어떤 실행을 유발하지 않는다. 정적 타입 체커(mypy, pyright)를 별도로 돌려야 이상을 감지한다.
Cell 1Markdown
FastAPI에서는 사용자가 정의한 클래스 자체를 타입으로 사용합니다. 특히 데이터 검증 라이브러리인 Pydantic 모델을 타입 힌트로 지정하면, FastAPI가 자동으로 JSON 데이터를 파싱하고 검증해줍니다.
dataclass의 한계를 지적하고, 해결책으로 Pydantic BaseModel을 도입한다는 전환.
Cell 2 — Item 모델과 FastAPI 친화 함수Code
from pydantic import BaseModel

class Item(BaseModel):
    name: str
    price: float
    is_offer: str | None = None   # 원본은 bool | None

# FastAPI에서 이런 식으로 사용하게 됩니다.
def create_item(item: Item):
    return {"item_name": item.name, "item_price": item.price}
무엇을: BaseModel을 상속한 Item 클래스를 정의. 그 타입을 받는 create_item 함수를 준비.
왜: FastAPI에서 엔드포인트 파라미터로 Pydantic 모델을 쓰는 실전 모양을 먼저 보여준다. 네트워크 없이도 순수 파이썬으로 동일한 인터페이스가 테스트 가능하다는 점이 중요하다.
유니온 문법 X | None은 파이썬 3.10+의 줄임 표현이고, Union[X, None] 또는 Optional[X]와 같다. 셋은 완전히 동일한 타입을 의미한다.
Cell 3 — 모델 인스턴스 생성 → 함수 호출Code
create_item(item=Item(name="Foo", price=50.2))
무엇을: Item 인스턴스를 생성해 create_item에 넣는다. 결과는 {'item_name': 'Foo', 'item_price': 50.2}.
왜: 모델 생성이 즉시 검증을 수행함을 강조한다. name, price가 빠지거나 잘못된 타입이면 이 줄에서 예외가 터진다. FastAPI에서는 같은 로직이 HTTP 422로 전환된다.
Cell 4 — PydanticUser: 정상 · 강제 변환 · 실패Code
from pydantic import ValidationError

class PydanticUser(BaseModel):
    """타입 힌트를 사용하며, 런타임에 유효성 검사 및 강제 변환 수행."""
    id: int
    name: str
    is_active: bool = True

# (1) 올바른 데이터
p_user_data = PydanticUser(id=1, name="Bob")
print(f"✅ 유효한 데이터 (정상): {p_user_data}")

# (2) 타입 강제 변환
p_user_coerce = PydanticUser(id='2', name="Charlie")
print(f"🔄 타입 강제 변환 (정상): {p_user_coerce}")

# (3) 유효성 검사 실패 (bool 아님)
try:
    PydanticUser(id=3, name="David", is_active="not_a_bool")
except ValidationError as e:
    print(f"❌ 유효성 검사 실패: \n{e.errors()[0]}")
무엇을: 세 가지 시나리오를 연속 시연.
  1. 정상: 타입 일치 → 그대로 저장.
  2. 강제 변환: id='2' (문자열) → Pydantic이 자동으로 int(2)로 변환. 출력 객체의 id는 진짜 정수 2.
  3. 실패: is_active="not_a_bool" → bool로 해석 불가능하므로 ValidationError. e.errors()[0]은 어느 필드가 왜 잘못됐는지 구조화된 딕셔너리로 알려준다.
왜 이렇게 쓰는지: 실무에서는 "숫자로 들어와야 하는 값이 문자열로 도착"하는 일이 흔하다. Pydantic은 안전하게 해석 가능한 변환은 허용하고, 애매하거나 위험한 입력은 거절하는 두 축으로 동작한다.
Cell 5Code (빈 셀)
학생 연습 공간. 자기 데이터 구조로 BaseModel을 만들어 보라는 의도.

4. 타입 강제 변환(coercion)의 규칙

"무엇을 허용하고 무엇을 거절하는가"가 자주 질문된다. 경험적 규칙:

  • int: 숫자 문자열('42'), True/False(→1/0) 허용. '42.5', 'abc' 거절.
  • float: 숫자 문자열 허용, int 허용. 'abc' 거절.
  • bool: True/False와 관습 문자열('true'/'false'/'1'/'0'/'yes'/'no') 허용. "not_a_bool"은 거절 → 본 셀의 (3)이 이 규칙에 걸려 실패한다.
  • str: 기본적으로 다른 타입을 받지 않는다(엄격). v1에서는 허용했지만 v2에서 엄격해졌다.
  • datetime: ISO 8601 문자열('2026-04-17T12:34') 자동 파싱.
엄격 모드 의도와 다른 변환을 원천 차단하려면 model_config = ConfigDict(strict=True)를 두거나 필드별 Strict 타입을 쓴다. 금융·인증 관련 값은 엄격 모드를 권장.

5. ValidationError 구조

e.errors()는 리스트이며, 각 원소는 아래 키로 구성된다.

[
  {
    "type": "bool_parsing",          # 에러 카테고리
    "loc":  ("is_active",),           # 문제가 된 경로(중첩 모델이면 깊어짐)
    "msg":  "Input should be a valid boolean ...",
    "input": "not_a_bool",             # 실제 들어온 값
    "url":  "https://errors.pydantic.dev/..."
  }
]

프런트엔드에 응답할 때 loc을 활용해 "어느 필드가 왜 실패했는지"를 화면에 정확히 매핑할 수 있다. FastAPI의 422 응답이 바로 이 구조를 감싼다.

6. FastAPI와의 연결

Pydantic 없이

def create_user(request):
    body = json.loads(request.body)
    # 수동 검증
    if not isinstance(body.get('id'), int):
        return 400, ...
    if not body.get('name'):
        return 400, ...
    ...

Pydantic + FastAPI

@app.post("/users")
def create_user(user: PydanticUser):
    # 여기 도달하면 user는 이미 검증·변환 완료
    return user

검증·문서화·IDE 자동완성·OpenAPI 스키마가 모두 같은 모델로부터 파생된다. "선언이 곧 문서이자 검증기이자 파서"라는 구조가 FastAPI 생산성의 비밀이다.

7. 심화: Field, validator, model_dump

from pydantic import BaseModel, Field, field_validator

class Product(BaseModel):
    sku: str = Field(pattern=r"^[A-Z]{3}-\d{4}$")   # 정규식 제약
    price: float = Field(gt=0, le=1_000_000)       # 0 초과 100만 이하
    tags: list[str] = Field(default_factory=list, max_length=5)

    @field_validator("sku")
    @classmethod
    def upper_only(cls, v: str) -> str:
        return v.upper()

p = Product(sku="abc-0001", price=42.5, tags=["new"])
p.model_dump()        # dict 변환
p.model_dump_json()   # 문자열 JSON 변환
  • Field: 제약(gt/ge/lt/le, min_length/max_length, pattern), 기본값 팩토리, alias, 설명(description) 등을 부여한다.
  • field_validator: 필드 단위 후처리/정규화를 선언한다. 이 예시는 SKU를 대문자로 통일.
  • model_dump / model_dump_json: v2의 직렬화 공식 메서드. v1의 dict()/json()은 이름이 바뀌었다.

8. 함정 3가지

  1. dataclass의 안정감: dataclass만 쓰다 "안 터지니 괜찮다"고 생각하기 쉽지만, 외부 입력을 다루는 순간 검증 부재는 잠재적 결함이 된다. 영역이 확장되면 Pydantic으로 옮기는 것이 안전.
  2. v1 문법 혼용: 인터넷 예제 중 @validator, .dict(), parse_obj는 v1 API. v2에서는 각각 @field_validator, .model_dump(), .model_validate()로 바뀌었다.
  3. bool 강제 변환의 의외성: is_active=1은 통과해 True가 된다. 숫자 0/1을 불리언으로 받는 게 곤란하면 StrictBool 또는 strict=True를 써야 한다.

9. 자체 점검 문제

  1. DataUser(id=2, name=123)이 에러를 내지 않는 이유를 한 문장으로.
  2. PydanticUser(id='2', name='Charlie')이 성공하는 이유를 타입 강제 변환 관점에서 설명.
  3. ValidationError.errors()[0]의 loc, type, input 키가 각각 어떤 정보를 담는지.
  4. Pydantic 모델의 인스턴스를 JSON 문자열로 바꾸려면 어떤 메서드를 쓰나?
  5. Field(gt=0, le=1_000_000)가 부여하는 제약을 평어로.
한 줄 요약 타입 힌트만 쓰고 실행시간 보증이 필요하면 Pydantic. 단순 컨테이너로 코드 안쪽에서만 도는 값이면 dataclass로 충분.
