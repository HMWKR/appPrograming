1. 왜 타입 힌트가 FastAPI 코드 80%의 열쇠인가

파이썬은 동적 타입 언어이지만 3.5 이후 PEP 484로 선택적 타입 힌트를 공식 지원한다. 실행에는 영향을 주지 않지만 ① IDE 자동완성, ② 정적 분석기(mypy/pyright) 경고, ③ Pydantic·FastAPI·typer·pydantic-settings 같은 타입 기반 런타임 도구에 정보를 제공한다. FastAPI 함수의 시그니처를 읽는다는 것은 곧 타입 힌트를 읽는다는 것이다.

한 줄 타입 힌트는 "실행 시 강제되지 않는 약속"이지만, Pydantic·FastAPI가 그 약속을 실제 검증기로 승격시킨다.

2. 기본: 변수·함수 (셀 0–6)

Cell 0 · 1Markdown
변수: name: str = "Gemini"
함수: def 함수명(인자: 타입) -> 반환타입:
핵심 문법 두 줄을 먼저 머리에 박는다. 변수는 이름 : 타입 = 값, 함수는 def f(x: T) -> R:. 이 패턴이 노트북 전체를 관통한다.
Cell 2 — 첫 함수Code
def greeting(name: str) -> str:
    return f"Hello, {name}"

greeting(name='Gemini')
무엇을: 문자열을 받아 문자열을 돌려주는 단순 함수.
왜 이렇게 쓰는지: -> str은 "반환값이 문자열"이라는 명세다. IDE는 이 정보를 보고 호출부에서 .upper() 같은 문자열 메서드를 자동완성으로 제안한다.
Cell 3 — float 입력 · str 반환Code
def surface_area_of_cube(edge_length: float) -> str:
    return f"The surface area of the cube is {6 * edge_length ** 2}."
무엇을: 한 변 길이를 받아 표면적을 문장 문자열로 반환.
포인트: 반환 타입이 float이 아니라 str이라는 점. 반환 형식이 함수명과 타입에서 곧바로 드러나야 한다는 설계 원칙.
Cell 4 · 5 — 함수 호출 두 가지Code
surface_area_of_cube(edge_length=4.00)   # 키워드 인자
surface_area_of_cube(4.00)                # 위치 인자
왜 두 번 보여주는가: 타입 힌트는 호출 방식과 독립적이다. 위치로 넘기든 키워드로 넘기든 타입 체커는 같은 규칙을 적용한다.
Cell 6 — 반환 타입 실제 확인Code
res = surface_area_of_cube(4.00)
type(res)
무엇을: 실제로 반환된 객체의 런타임 타입을 확인 → str.
런타임은 검사하지 않는다 힌트를 -> int로 바꾸고 실제로 str을 반환해도 파이썬은 에러를 내지 않는다. 타입 힌트의 검사는 외부 도구 또는 Pydantic이 대신해 준다.

3. 컬렉션 타입 (셀 7–13)

Cell 7Markdown
Python 3.9 버전부터는 기본 list, dict 키워드를 그대로 사용합니다.
3.8까지는 typing.List[int]처럼 대문자를 써야 했지만, 3.9+에서는 list[int]로 충분하다.
Cell 8 — 타입 별칭 type VectorCode
type Vector = list[float]

def scale(scalar: float, vector: Vector) -> Vector:
    return [scalar * num for num in vector]

new_vector = scale(2.0, [1.0, -4.2, 5.4])
무엇을: list[float]에 Vector라는 의미 있는 이름을 부여하고, 함수 시그니처에 그대로 재사용.
왜: 수학적 도메인 개념("벡터")이 코드에서도 드러난다. 반복적으로 등장하는 복잡한 타입을 별칭으로 만들면 유지보수에 유리. type Vector = ...는 PEP 695 (Python 3.12+)의 문법. 구버전에서는 Vector = list[float]만으로도 동일한 효과.
Cell 9 — scale 호출 결과Code
new_vector
출력: [2.0, -8.4, 10.8].
Cell 10 — list/dict/tuple 기본 선언Code
names: list[str] = ["Alice", "Bob", "Charlie"]
user_scores: dict[str, int] = {"Alice": 100, "Bob": 90}
user_info: tuple[int, str] = (1, "Admin")
포인트:
  • list[str]: 원소의 타입이 문자열. 길이는 제한 없음.
  • dict[str, int]: 키는 문자열, 값은 정수.
  • tuple[int, str]: 순서가 있는 정확히 2개의 원소. 첫째 int, 둘째 str.
tuple과 list의 큰 차이 list는 "같은 종류 원소가 여러 개", tuple은 "서로 다른 역할을 가진 고정 길이". DB 한 행((id, name, active))을 표현할 땐 tuple, 사용자 이름 목록엔 list.
Cell 11 · 12 · 13 — 확인Code
names         # ['Alice', 'Bob', 'Charlie']
user_scores   # {'Alice': 100, 'Bob': 90}
user_info     # (1, 'Admin')
각 변수의 실제 내용을 출력해 타입 선언과 값이 일치함을 육안 확인.

4. Optional · Union · | (셀 14–21)

Cell 14Markdown
Union: 여러 타입 중 하나일 수 있음 · Optional: 해당 타입이거나 None임 (사실 Union[타입, None]의 줄임표현)
Cell 15 — |와 Union 혼용Code
from typing import Union

# Python 3.10+ 권장 방식
def get_item(item_id: int, description: str | None = None):
    return {"id": item_id, "desc": description}

# 3.10 미만 버전 방식
def get_user(user_id: Union[int, str]):
    return {"id": user_id}
무엇을: 두 함수 시그니처로 유니온의 두 가지 작성법을 나란히 제시.
  • str | None = None: "문자열 또는 없음"이라는 의미. 기본값이 있어 선택 인자가 된다.
  • Union[int, str]: |의 구버전 호환 표현. 세미 공식 표준 라이브러리 typing에서 import.
실무 권장 3.10+ 환경이면 |를 쓰는 쪽이 간결하다. 구버전 호환이 필요하면 from __future__ import annotations를 파일 상단에 두면 3.7+에서도 str | None 문법이 "문자열로만" 파싱돼 동작한다.
Cell 16 · 17 — get_item 호출Code
get_item(item_id=1)                    # {'id': 1, 'desc': None}
get_item(item_id=1, description='test') # {'id': 1, 'desc': 'test'}
포인트: 선택 인자 description은 생략해도 되고, 값을 주면 그대로 들어간다. FastAPI 엔드포인트에서 query string의 필수/선택 여부가 이 문법으로 자연스럽게 표현된다.
Cell 18 · 19 — get_user 호출Code
get_user(user_id=1)      # {'id': 1}  — int 허용
get_user(user_id='1')    # {'id': '1'} — str도 허용
관찰: Union[int, str]이 있어 둘 다 유효. 단, 함수 자체는 어떤 검증도 하지 않는다. "어떤 타입이든 그대로 통과"라는 "약속"의 선언일 뿐.
Cell 20 — Optional[int]Code
from typing import Optional

def greeting(name: str, enthusiasm: Optional[int] = None) -> str:
    if enthusiasm is None:
        return f"Hello, {name}."
    return f"Hello, {name}!" * enthusiasm
동치 관계: Optional[int] ≡ Union[int, None] ≡ int | None. 셋 모두 "정수 또는 None". 가독성을 위해 "의미상 None이 유효한 값"임을 강조하고 싶을 때 Optional을 선호하는 관행이 있다.
Cell 21 — 결과 확인Code
greeting_result = greeting("Alice", 3)
print(greeting_result)   # Hello, Alice!Hello, Alice!Hello, Alice!
관찰: 문자열 반복 연산자 *가 3번 곱해져 출력. 열정치(enthusiasm)를 None으로 주면 단문이 된다.

5. 제네릭 컬렉션·Mapping·가변길이 튜플 (셀 22–23)

Cell 22 — 복합 타입 시그니처Code
from typing import List, Tuple
from collections.abc import Mapping

def process_records(
    data: List[Tuple[str, int]],                    # 리스트[(str,int)]
    settings: Mapping[str, Union[str, int]],            # dict 대신 Mapping
    coordinates: Tuple[float, float, float]              # 고정 길이 3
) -> Tuple[int, ...]:                                     # 가변 길이 튜플
    print(f"데이터: {data}, 데이터 타입 {type(data)}")
    processed_count = len(data)
    # ... (설정·좌표 출력 생략)
    return (processed_count)   # ⚠ 소괄호만으로는 튜플이 안 된다
무엇을: 세 종류의 입력을 묶어 다양한 제네릭 표기를 한 자리에서 보여 주는 종합 예.
왜 Mapping인가: dict를 쓰면 "딱 dict만 받는다"는 뜻이 된다. Mapping은 "딕셔너리처럼 동작하는 모든 것"을 허용하는 추상 타입이다. 표준 dict, OrderedDict, MappingProxyType 등이 자연스럽게 들어갈 수 있어 API 유연성이 높아진다.
Tuple[int, ...]의 의미: 앞의 Tuple[int, str]처럼 정확히 두 개가 아니라, "임의 개수의 정수" 튜플. ...(Ellipsis)가 "개수 가변"을 뜻한다.
중요한 함정 본 함수는 return (processed_count)로 끝나는데, 이건 튜플이 아니라 그냥 정수다. 1-튜플을 만들려면 반드시 (processed_count,)처럼 쉼표가 있어야 한다. 반환 타입 선언은 Tuple[int, ...]인데 실제 반환이 int라서 타입 체커가 경고하거나, 호출부에서 튜플로 사용하면 런타임 오류가 난다.
Cell 23 — 실행Code
processed = process_records(records, settings, coordinates)
무엇을: 노트북 상단에서 정의한 세 입력을 실제로 전달해 함수를 실행. 출력 문장은 각 파라미터의 타입을 확인시켜 준다.

6. FastAPI에서 타입 힌트가 중요한 이유 (셀 24)

Cell 24Markdown
① 자동 데이터 검증: age: int로 선언했는데 "twenty"가 오면 FastAPI가 422 에러를 생성.
② 자동 문서화(Swagger): .../docs 에 API 문서가 자동 생성.
③ 에디터 지원: 자동 완성이 완벽하게 작동.
왜 이 세 가지인가: 타입 힌트 1줄이 3가지 도메인(런타임 검증, 문서, IDE)을 동시에 개선한다. 이것이 "한 번 쓰면 세 번 덕본다"는 FastAPI 특유의 생산성이다.
Cell 25Code (빈 셀)
학생이 FastAPI 엔드포인트 스텁을 직접 작성하도록 비운 자리.

7. 심화: 타입 별칭, Literal, Protocol, Self

7-1. 타입 별칭 (Type Alias)

# PEP 695 (3.12+)
type UserId = int
type Row = tuple[UserId, str]

# 3.12 미만 호환
UserId = int
Row = tuple[UserId, str]

도메인 언어로 타입을 바꾸면 함수 시그니처 한 줄을 읽고 의미가 바로 들어온다.

7-2. Literal

from typing import Literal

def set_status(s: Literal["active", "paused", "closed"]) -> None: ...

가능한 값의 집합이 작을 때 열거형보다 가볍게 쓸 수 있다. FastAPI는 이걸 OpenAPI의 enum으로 변환한다.

7-3. Protocol (구조적 서브타이핑)

from typing import Protocol

class SupportsLen(Protocol):
    def __len__(self) -> int: ...

def first_half(x: SupportsLen) -> int:
    return len(x) // 2

"특정 메서드만 있으면 된다"를 표현한다. 상속 관계가 아니라 모양으로 맞으면 허용 — 파이썬다운 덕 타이핑을 타입 시스템으로 옮겨 놓은 개념.

7-4. Self (PEP 673, 3.11+)

from typing import Self

class Counter:
    def incr(self) -> Self:   # 자기 자신 타입을 정확히
        ...
        return self

서브클래스에서도 타입이 올바르게 좁혀진다. 체이닝 API를 만들 때 유용.

8. 함정과 자주 묻는 질문

  • 단일 원소 튜플: (x,) 쉼표 필수. (x)는 그냥 괄호.
  • Tuple[int, ...] vs tuple[int, ...]: 전자는 typing, 후자는 3.9+의 내장. 동치.
  • Optional은 "반드시 null 가능"이 아니다: Optional[int] = None처럼 기본값까지 줘야 "생략 가능"이 된다. 기본값이 없는 Optional[int]는 "호출 시 반드시 int나 None 중 하나를 넘겨야 한다"는 뜻.
  • 실행시 검사 없음: 타입이 맞지 않아도 파이썬은 그냥 돌아간다. mypy/pyright를 CI에 넣거나, Pydantic 모델로 검증 지점을 모아야 실효성이 생긴다.
  • Mapping vs dict: 함수 매개변수는 Mapping(읽기만 필요할 때), 반환은 dict(수정 가능)가 전형적인 가이드라인.
