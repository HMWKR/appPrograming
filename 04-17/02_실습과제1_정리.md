## 4. 테이블 스키마

```sql
CREATE TABLE customers (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    username  TEXT NOT NULL UNIQUE,
    full_name TEXT
)
```

- `AUTOINCREMENT` — id는 자동 증가. 삭제 후에도 번호 재사용하지 않음.
- `UNIQUE` — 같은 username 재삽입 시 `IntegrityError`. Gradio 앱은 `INSERT OR IGNORE`로 **방어적 처리**.

## 5. 강의 예제(Cell 13) 핵심 해설

### 5-1. init_customer_db
```python
if os.path.exists(CUSTOMER_DB_NAME):
    os.remove(CUSTOMER_DB_NAME)
conn = sqlite3.connect(CUSTOMER_DB_NAME)
```
- **무엇을**: 이전 DB 파일을 지우고 새로 연결.
- **왜**: 과제 시연 목적상 매 실행마다 **깨끗한 상태**를 보장하기 위함. 실제 운영 환경에서는 지우지 않는다.

### 5-2. Bulk Insert
```python
customer_data = df_top10.apply(lambda row: (row['고객ID'], row['고객ID']), axis=1).tolist()
cursor.executemany(
    "INSERT INTO customers (username, full_name) VALUES (?, ?)",
    customer_data,
)
```
- **무엇을**: DataFrame을 튜플 리스트로 변환 후 executemany로 한 번에 삽입.
- **왜 `?` 바인딩**: 문자열 포매팅(`f"..."`) 사용 시 **SQL 주입** 위험. `?`는 드라이버가 안전하게 이스케이프한다.

### 5-3. Update / Delete의 `WHERE`
```python
cursor.execute("UPDATE customers SET full_name = ? WHERE username = ?", (new_name, target_user))
cursor.execute("DELETE FROM customers WHERE username = ?", ('CUST_0003',))
```
- `WHERE` 누락 시 **테이블 전체가 수정·삭제**된다 — 운영 시 반드시 테스트 환경에서 선행 검증.

### 5-4. finally + conn.close()
```python
finally:
    if conn:
        conn.close()
```
- 예외가 발생해도 커넥션을 닫아 **파일 핸들 누수를 방지**한다. 더 안전한 대안은 `with sqlite3.connect(...) as conn:` 컨텍스트 매니저.

## 6. Gradio 앱 설계 (`task1_gradio_app.py`)

### 6-1. UI 구조 — 5개 탭
| 탭 | 기능 | 입력 | 출력 |
|---|---|---|---|
| ① 초기화 & 적재 | DB 재생성 + CSV 상위 N개 삽입 | CSV URL, N | 로그 + 현재 테이블 |
| ② Read | 전체 조회 / username 검색 | username | 로그 + 테이블 |
| ③ Create | 신규 고객 추가 | username, full_name | 로그 + 테이블 |
| ④ Update | full_name 수정 | username, 새 full_name | 로그 + 테이블 |
| ⑤ Delete | 고객 삭제 | username | 로그 + 테이블 |

### 6-2. 설계 의도

- **연결을 매 호출마다 새로 연다(`connect()`).** Gradio는 요청을 여러 스레드에서 처리하는데, `sqlite3.Connection`은 기본적으로 생성 스레드에서만 사용 가능하기 때문이다.
- **모든 CRUD 함수가 현재 테이블 DataFrame을 반환**한다 → 사용자가 조작 직후 결과를 즉시 확인 가능.
- **파라미터 바인딩(`?`) 일관 사용** — SQL 주입 완전 차단.
- **`INSERT OR IGNORE`** — UNIQUE 제약 충돌이 UI를 멈추지 않도록 방어.
- **`Path(__file__).resolve().parent / "customer_data.db"`** — DB 파일이 항상 앱과 같은 폴더에 생성되도록 절대 경로화. 임의 cwd에서 실행되어도 경로 안정.

### 6-3. CRUD 검증 로그 (실행 결과 `[검증됨]`)

```
[init_and_load] [OK] 테이블 재생성 완료. 10/10개 삽입됨.
[read]          [OK] 조회 결과: id=2, username=CUST_0002, full_name=CUST_0002
[create]        [OK] 'CUST_TEST' 추가 완료.
[update]        [OK] 'CUST_0002'의 full_name 수정 완료 (1행).
[delete]        [OK] 'CUST_0003' 삭제 완료 (1행).
```

최종 테이블: 10행(초기) + 1행(CUST_TEST) - 1행(CUST_0003) = **10행**. CUST_0002 의 full_name 이 `Updated User 02` 로 변경됨.
