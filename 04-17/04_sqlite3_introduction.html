1. 왜 SQLite3로 시작하는가

  • 무설정: Python 표준 라이브러리. pip install 불필요.
  • 단일 파일 DB: 서버 프로세스가 없고 .db 파일 하나가 곧 데이터베이스.
  • 표준 SQL: CREATE/INSERT/SELECT/UPDATE/DELETE의 기본 문법이 PostgreSQL·MySQL과 거의 동일.
  • 가벼우나 실전감: 브라우저, 스마트폰, 비행기 제어, Firefox 북마크까지 모두 SQLite다. 학습용이면서 실전용.
이 노트북의 학습 흐름 ① 파일 DB 연결 → ② CREATE TABLE → ③ C/R/U/D 연습 → ④ 실습과제 1 (CSV→DB) → ⑤ 실습과제 2 (모듈 리팩토링).

Cell 1Markdown
sqlite3.connect()는 파일이 없으면 자동으로 생성 · cursor는 SQL 실행 객체
Connection(연결)과 Cursor(실행 창구)의 역할 분리를 미리 알려 준다. Connection은 트랜잭션/열기·닫기 담당, Cursor는 execute·fetch 담당.
Cell 2 — init_db 함수Code
import sqlite3
import os

DB_NAME = "test_sqlite3.db"

def init_db():
    if os.path.exists(DB_NAME):
        os.remove(DB_NAME)
    conn = sqlite3.connect(DB_NAME)
    cursor = conn.cursor()
    cursor.execute("""
        CREATE TABLE users (
            id       INTEGER PRIMARY KEY,
            name     TEXT NOT NULL,
            fullname TEXT
        )
    """)
    conn.commit()
    return conn
무엇을: 기존 DB 파일 삭제 → 새 연결 → users 테이블 생성 → commit → 연결 객체 반환.
왜 파일부터 지우는가: 노트북은 재실행을 전제로 한다. CREATE TABLE users는 이미 테이블이 있으면 에러를 내므로, 가장 단순한 멱등성 보장이 "파일 삭제 후 재생성"이다.
실무에서는? 실제 서비스에서 os.remove는 데이터를 날리는 파괴적 연산이다. 대신 CREATE TABLE IF NOT EXISTS를 쓰거나, 마이그레이션 도구(Alembic/Flyway)로 스키마 버전 관리를 한다.
Cell 3Markdown
executemany()는 리스트 데이터를 한 번에 여러 행으로 삽입합니다.
Cell 4 — 벌크 INSERTCode
conn = init_db()
cursor = conn.cursor()

users_data = [
    ('spongebob', 'Spongebob Squarepants'),
    ('patrick',   'Patrick Star'),
    ('squidward', 'Squidward Tentacles'),
]

cursor.executemany(
    "INSERT INTO users (name, fullname) VALUES (?, ?)",
    users_data
)
conn.commit()
print(f"{len(users_data)}개의 사용자 데이터 삽입 완료.")
무엇을: 튜플 리스트를 한 번의 호출로 모두 삽입.
왜 executemany인가: 루프 for row in rows: cur.execute(...)보다 빠르다. 드라이버가 내부적으로 배치 처리하고 트랜잭션 지점을 최적화한다. 대량 적재일수록 체감이 크다.
왜 ? 바인딩인가: 문자열 포매팅으로 값을 박아 넣으면 SQL 주입이 가능하다. name = 'x'; DROP TABLE users;-- 같은 입력이 쿼리 자체를 변형시킨다. ?는 값만 전달돼 절대 쿼리 구조를 바꾸지 못한다.
Cell 5Markdown
fetchall()은 모든 행을 리스트로, fetchone()은 첫 번째 행 하나만 반환 · 결과는 튜플 형식이며 인덱스로 각 컬럼에 접근
Cell 6 — 전체 조회Code
cursor.execute("SELECT id, name, fullname FROM users")
rows = cursor.fetchall()

for row in rows:
    print(f"ID: {row[0]}, Name: {row[1]}, FullName: {row[2]}")
포인트: SELECT * 대신 SELECT id, name, fullname처럼 컬럼 순서를 명시. 이렇게 하면 나중에 스키마에 컬럼이 추가되어도 인덱싱이 어긋나지 않는다.
튜플 → dict 변환 꿀팁 conn.row_factory = sqlite3.Row를 설정하면 row['name']처럼 이름으로 접근 가능. 컬럼 순서에 의존하지 않아 훨씬 견고하다.
Cell 7 — 조건 조회 + fetchoneCode
cursor.execute(
    "SELECT * FROM users WHERE name = ?",
    ('patrick',)   # 튜플 주의: 쉼표 필수
)
patrick = cursor.fetchone()
print(patrick)
왜 쉼표가 필수인가: ('patrick')은 그냥 문자열이다. ('patrick',)처럼 쉼표가 있어야 1-튜플이 되어 executor가 "파라미터 1개"라고 인식한다.
fetchone vs fetchall: fetchone은 첫 행만, 없으면 None. 로그인 같은 "단일 일치" 조회에서 필수 패턴.
Cell 8Markdown
WHERE 조건을 빠뜨리면 테이블 전체가 수정되니 항상 확인하세요.
Cell 9 — UPDATECode
new_fullname = 'S. Squarepants'

cursor.execute(
    "UPDATE users SET fullname = ? WHERE name = ?",
    (new_fullname, 'spongebob')
)
conn.commit()

cursor.execute("SELECT * FROM users WHERE name = 'spongebob'")
print(cursor.fetchone())
포인트: UPDATE 뒤 반드시 commit(). 이어서 SELECT로 사후 검증하는 습관을 보여준다.
WHERE 누락의 대재앙 UPDATE users SET fullname='X' (WHERE 없음)은 모든 행의 fullname을 X로 만든다. 운영 DB에서는 치명적. "UPDATE·DELETE는 먼저 SELECT ... WHERE로 영향 행 수를 확인"이 철칙.
Cell 10Markdown
WHERE 없이 실행하면 테이블의 모든 데이터가 삭제됩니다.

cursor.execute(
    "DELETE FROM users WHERE name = ?",
    ('squidward',)

conn.commit()

구조: UPDATE와 동일한 패턴. 파라미터 바인딩 → commit → 재조회 검증.

Cell 13 — 실습과제 1 예제 코드Code
import pandas as pd
import sqlite3, os

CUSTOMER_DB_NAME = "customer_data.db"

def init_customer_db():
    if os.path.exists(CUSTOMER_DB_NAME):
        os.remove(CUSTOMER_DB_NAME)
    conn = sqlite3.connect(CUSTOMER_DB_NAME)
    cursor = conn.cursor()
    cursor.execute("""
        CREATE TABLE customers (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
            username TEXT NOT NULL UNIQUE,
            full_name TEXT
        )
    """)
    conn.commit()
    return conn

def perform_customer_crud_operations(conn):
    cursor = conn.cursor()
    csv_url = "https://raw.githubusercontent.com/ancestor9/data/main/customers.csv"

    # [C] CSV 상위 10개 삽입
    df = pd.read_csv(csv_url).head(10)
    customer_data = df.apply(lambda row: (row['고객ID'], row['고객ID']), axis=1).tolist()
    cursor.executemany("INSERT INTO customers (username, full_name) VALUES (?, ?)", customer_data)
    conn.commit()

    # [R] 상위 10개 조회
    cursor.execute("SELECT id, username, full_name FROM customers LIMIT 10")
    for row in cursor.fetchall():
        print(row)

    # [U] CUST_0002 수정
    cursor.execute("UPDATE customers SET full_name = ? WHERE username = ?", ('Updated User 02', 'CUST_0002'))
    conn.commit()

    # [D] CUST_0003 삭제
    cursor.execute("DELETE FROM customers WHERE username = ?", ('CUST_0003',))
    conn.commit()

if __name__ == "__main__":
    conn = None
    try:
        conn = init_customer_db()
        perform_customer_crud_operations(conn)
    finally:
        if conn: conn.close()
핵심 설계:
  • AUTOINCREMENT: id를 자동 증가. 삭제된 id를 재사용하지 않으므로 참조 무결성 추적에 유리.
  • UNIQUE username: 같은 고객ID가 두 번 들어가지 못하게 강제.
  • pd.read_csv(url).head(10): 원격 CSV를 즉시 DataFrame으로 로드 후 상위 10개만 취함. 시연 목적에 적합한 크기.
  • .apply(lambda row: (...), axis=1).tolist(): DataFrame → 튜플 리스트로 바꾸는 관용구. executemany의 입력 형식에 맞춘다.
  • finally: conn.close(): 예외가 나도 연결을 확실히 닫아 파일 핸들 누수 방지.
더 짧은 방법 df.to_sql("customers", conn, if_exists="append", index=False)는 pandas가 자체적으로 벌크 INSERT를 실행해 준다. 다만 학습 목적으로는 executemany를 먼저 경험한 뒤 to_sql을 알게 되는 순서가 좋다.

6. 보안·트랜잭션·동시성 심화

6-1. SQL 주입과 파라미터 바인딩

# 위험: 주입 가능
name = request.get("name")   # "'; DROP TABLE users;--"
cur.execute(f"SELECT * FROM users WHERE name='{name}'")

# 안전: 드라이버가 값으로만 취급
cur.execute("SELECT * FROM users WHERE name = ?", (name,))

6-2. 트랜잭션 · 컨텍스트 매니저

with sqlite3.connect(DB) as conn:
    conn.execute("INSERT ...")
    conn.execute("UPDATE ...")
# with 블록이 정상 종료되면 자동 COMMIT, 예외 발생 시 ROLLBACK

본 노트북은 명시적 commit을 쓰지만, 실무에서는 with 블록이 원자성 보장을 쉽게 해 준다.

6-3. 동시성

SQLite는 파일 락으로 동시성을 관리한다. 여러 스레드/프로세스에서 쓸 때 주의점:

  • 한 Connection은 만든 스레드에서만 사용(기본). 다른 스레드에서 쓰려면 check_same_thread=False 또는 스레드별 새 연결.
  • 대량 쓰기 충돌을 줄이려면 WAL 모드: PRAGMA journal_mode=WAL;
  • 고동시성이 필요하면 SQLite가 적합하지 않다 — PostgreSQL·MySQL로 이동.

7. 실무 함정 10가지

  1. commit 누락: execute만 하고 commit을 잊으면 파이썬 종료 시 롤백.
  2. WHERE 누락: UPDATE/DELETE 재난의 1위.
  3. SELECT *: 스키마 확장 시 인덱스 기반 접근이 깨진다.
  4. f-string SQL: SQL 주입 창구.
  5. 1-튜플: (x,) 쉼표 필수. 없으면 문자열.
  6. 경로 상대화: "db.sqlite"는 cwd 의존. Path(__file__).parent / "db.sqlite"로 고정.
  7. 연결 누수: finally: conn.close() 또는 with.
  8. 대량 executemany를 여러 트랜잭션으로 쪼개는 실수: 중간 commit이 끼면 전부 느려진다. 수만 건은 한 트랜잭션으로.
  9. Row Factory 무시: row[0] 대신 row['id']로 바꾸면 유지보수가 쉬워진다.
  10. 동일 스레드 가정: Gradio/FastAPI 같은 멀티스레드에서 Connection을 공유하면 크래시. 함수마다 새로 connect하거나 풀을 사용.

  2. UPDATE users SET name='X'를 실수로 실행했을 때의 복구 방법 두 가지.
  3. 파라미터 바인딩 없이 문자열 포매팅으로 SQL을 만들면 생길 수 있는 공격 한 가지를 예로.
  4. fetchone()이 None을 반환하는 조건은?
  5. Cell 13의 AUTOINCREMENT를 빼면 어떤 동작 차이가 생기는가?
  6. 실습과제 2에서 모듈을 init.py가 아니라 database_init.py로 명명하지 않은 이유를 설명.
