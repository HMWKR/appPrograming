# 06주차 실습과제 2 — 모듈화 리팩토링 정리

> 출처: `06week/week6/04_sqlite3_introduction.ipynb` **Cell 14**

---

## 1. 과제 원문

> vscode에서 `init.py`, `create.py`, `read.py`, `update.py`, `delete.py` 로 모듈화하여 실행하는 코드로 refactoring하라.

즉, **Cell 13의 단일 파이썬 블록**을 역할별 5개 파일로 쪼개고 한 곳에서 조합 실행하도록 구성하는 과제.

---

## 2. 구현 위치

```
review/task2_modules/
├── init.py      ← DB 연결·재설정·테이블 생성
├── create.py    ← INSERT (단건/일괄/CSV)
├── read.py      ← SELECT (전체/단건/카운트)
├── update.py    ← UPDATE full_name
├── delete.py    ← DELETE by username
└── main.py      ← 위 5개를 조합해 Cell 13 시나리오 재현
```

각 모듈은 `if __name__ == "__main__":` 로 **단독 실행**도 가능하다. 개발 중 모듈별로 따로 돌려 디버깅할 수 있도록 설계.

---

## 3. 책임 분할 이유

| 모듈 | 책임 | 왜 이렇게 나눴는가 |
|---|---|---|
| `init.py` | `connect / reset_db / create_table / init_db` | **연결 지점 단일화**: 다른 모듈이 `from init import connect` 한 줄로 연결을 얻는다. |
| `create.py` | `insert_one / insert_many / load_from_csv` | INSERT 의 3가지 입력 방식(단건·튜플 리스트·CSV URL)을 한 곳에 모음. |
| `read.py` | `fetch_all / fetch_by_username / count` | 부작용 없는 **순수 조회**만 모아 테스트 작성이 쉬움. |
| `update.py` | `update_full_name` | 과제 요구 단일 UPDATE 시나리오. 이름을 구체적으로 붙여 **WHERE 누락** 실수 방지. |
| `delete.py` | `delete_by_username` | UPDATE 와 동일한 취지. 범위를 `username` 로 명시. |
| `main.py` | 전체 시나리오 오케스트레이션 | 각 모듈을 import 해 순차 실행. 통합 테스트 역할. |

---

## 4. 주요 설계 결정

### 4-1. DB 경로 고정 (`Path(__file__).resolve().parent / "customer_data.db"`)

- 어느 폴더에서 실행되더라도 **모듈 파일과 같은 위치**에 DB 가 생성된다.
- 상대 경로 `"customer_data.db"` 는 cwd에 따라 달라져 혼란을 부른다 — 절대 경로화로 제거.

### 4-2. `create_table` 은 `CREATE TABLE IF NOT EXISTS`

- 강의 원본(Cell 13)은 `CREATE TABLE` 만 사용하고 **파일 삭제로** 중복 생성을 회피했다.
- 모듈화 후에는 `init.py` 단독 실행 시 기존 DB를 **보존한 채로** 테이블만 확인/생성할 수 있도록 `IF NOT EXISTS` 를 추가. `fresh=True` 플래그로 파일 재생성도 선택 가능.

### 4-3. `INSERT OR IGNORE` 기본값

- `insert_many` 는 UNIQUE 충돌 시 **무시** 하도록 `INSERT OR IGNORE`.
- 재실행(CSV 재로드)이 에러 없이 멱등적으로 동작하게 하기 위함. UNIQUE 제약이 여전히 `username` 중복을 막아 준다.

### 4-4. 반환값으로 영향 범위 전달

- `update.update_full_name`, `delete.delete_by_username` 은 `cur.rowcount` 를 반환.
- 호출부(main.py)가 **몇 행이 실제로 변경됐는지** 확인하며 진행 — "0행 변경"을 감지해 사용자 피드백 가능.

### 4-5. `from __future__ import annotations`

- 파이썬 3.9+ 의 제네릭 타입 힌트(`list[tuple]` 등)를 구버전 런타임에서도 파싱할 수 있게 해준다. 3주차 주제인 typehints 과 연결된 실무 감각.

## 5. 실행 결과 `[검증됨]`

```
[1/7] init_db 완료 → customer_data.db
[2/7] create.load_from_csv → 10개 시도

        (1, 'CUST_0001', 'CUST_0001')
        ...
        (10, 'CUST_0010', 'CUST_0010')
[4/7] read.fetch_by_username('CUST_0002') → (2, 'CUST_0002', 'CUST_0002')

        (2, 'CUST_0002', 'Updated User 02')
        (4, 'CUST_0004', 'CUST_0004')
        ...
        (10, 'CUST_0010', 'CUST_0010')
        총 9행
```

- Cell 13 의 최종 상태와 **정확히 일치** (10삽입 → 1수정 → 1삭제 → 9행 잔존).

## 6. 단독 실행 예시 (디버깅 시)

```bash
cd review/task2_modules

# DB 파일만 재생성
python init.py

# CSV 상위 10개 삽입 (init 이후)
python create.py

# 읽기
python read.py

# 수정 / 삭제
python update.py
python delete.py
```

각 모듈이 자기 책임 범위만 실행하므로 **부분 재현**이 가능하다.
